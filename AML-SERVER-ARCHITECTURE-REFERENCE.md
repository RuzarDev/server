## 1. Назначение системы

Backend для операционной автологистики: пользователи и роли, водители и статусы, транспорт и справочники, заявки на перевозки, техсобытия, расписания, ежедневные отчёты, уведомления, интеграции (в т.ч. внешние сервисы).

Стек основного API: **.NET 9**, **ASP.NET Core**, **EF Core**, **PostgreSQL**, **Hangfire**, **OpenTelemetry**, **Minimal API**, **Swagger/OpenAPI**.

---

## 2. Высокоуровневая структура репозитория

| Часть | Путь | Роль |
|--------|------|------|
| Основной API | `src/API/AML.API` | Бизнес-логика, vertical slices, CQRS, `Program.cs` как composition root |
| Отчёты | `src/Reports/AML.Reports` | Отдельное приложение (MVC/controllers, DevExpress), свой `DbContext` |
| Общие библиотеки | `src/Common/*` | Доменные типы, CQRS и pipeline, инфраструктура, абстракции presentation |
| Тесты | `tests/AML.API.Tests` | Сценарии по фичам и инфраструктуре, тестовый каркас |

**Важно:** в одном репозитории **две модели** — slice+CQRS+minimal endpoints в API и классические контроллеры в Reports. Решения для API не копируют слепо в Reports без учёта этого.

---

## 3. Архитектурный стиль `AML.API`

- **Vertical Slice Architecture**: код сгруппирован по фичам/доменам в `Features/<Модуль>/`, а не по техническим слоям «все контроллеры / все сервисы».
- **CQRS**: чтение через `IQuery<TResponse>` и `IQueryHandler<,>`, изменения через `ICommand` / `ICommand<TResponse>` и соответствующие handlers.
- **Minimal API + `IEndpoint`**: вместо классических контроллеров — классы с `MapEndpoint(IEndpointRouteBuilder)`, подключение через `AddEndpoints` / `MapEndpoints` и сборка из сборки API.
- **Общий каркас** в `AML.Common.Application`: сканирование handlers, декораторы (валидация, логирование), FluentValidation.

Типовой поток запроса:

1. HTTP → endpoint (`IEndpoint`).
2. Сборка `Query` / `Command`.
3. Вызов handler через DI.
4. Декораторы (валидация, логирование).
5. Работа с EF Core / доменными сервисами.
6. Ответ через единый подход к результату и маппинг в HTTP.

## 4. Структура фичи (соглашение)

Ориентир из `README` — для операции заводятся файлы в духе:

- `{Action}{Entity}.Endpoint.cs` — маршрут, метод, OpenAPI, права.
- `{Action}{Entity}.Query.cs` или `...Command.cs` — контракт операции.
- `{Action}{Entity}.QueryHandler.cs` или `...CommandHandler.cs` — логика.
- при необходимости `{Action}{Entity}.Validator.cs` — FluentValidation.

Имена и гранулярность смотри по соседним фичам в том же модуле — **первичный референс — живой код**, этот файл задаёт только рамку.

---

## 5. Данные и инфраструктура

- **PostgreSQL**, соглашение имён **snake_case** для БД.
- Два контекста: `ApplicationDbContext` (предметная область) и `ApplicationIdentityDbContext` (Identity). Миграции при старте API применяются из `Program.cs`.
- **Hangfire** — фоновые задачи; сценарии также в `Jobs/`.
- **OpenTelemetry**, health checks, CORS и прочее — собираются в `Program.cs` (известная зона роста: декомпозиция регистраций).

`Infrastructure/` — внешние интеграции, файловое хранилище, почта, уведомления, аудит (interceptor/middleware), настройки с шифрованием и т.д.

---

## 6. Тесты

`tests/AML.API.Tests`: зеркало `Features/<Модуль>`, плюс `Infrastructure/`, плюс `TestFramework/` (фабрики, фейки, SQLite и т.д.). Новая логика в стиле проекта сопровождается тестами на handler/сценарий там, где это уместно.

## 8. Как ты задаёшь вопросы (формат для обучения)

Чтобы ответ был максимально применимым к aml-server, в сообщении по возможности укажи:

1. **Цель** — что хочешь получить (новый endpoint, правка handler, миграция, джоб, интеграция).
2. **Модуль** — например `TransportationApplications`, `TechnicalEvents`, `Users`.
3. **Тип операции** — чтение (query) или изменение (command), или и то и другое.
4. **Ограничения** — обратная совместимость API, производительность, права, транзакции.
5. **Где смотреть** — если уже открыл конкретный файл или ветку, напиши путь или имя use case.

Короткие шаблоны:

- «Хочу добавить в `Features/X` операцию Y: вход Z, ответ W. Какие файлы завести и в каком порядке?»
- «Сравни два варианта (A/B) для этой задачи в духе aml-server и скажи, что выбрать и почему.»
- «Разбей на шаги: от контракта до теста, без лишнего.»

## 9. Как ведётся ответ (роль наставника)

При ответах по умолчанию:

- Опираться на **vertical slice + CQRS + `IEndpoint`** для `AML.API`, не предлагать «классический контроллер» без причины.
- Указывать **куда положить код** (папка фичи, нужен ли validator, трогается ли `Infrastructure`, нужна ли миграция).
- Объяснять **почему** в 1–3 предложениях, без воды: связь с существующими паттернами репозитория.
- При неоднозначности — дать **рекомендацию по умолчанию** и кратко назвать альтернативу.
- Для Reports явно отмечать, что там **другая** архитектурная модель.

Если вопрос общий («как учиться писать как в этом проекте»), ответ начинается с **одного маленького практического шага** (например: «открой фичу X и проследи путь от Endpoint до Handler»), затем — что смотреть дальше.
